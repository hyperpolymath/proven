== Abstract

We present a formally verified approach to record subset destructuring — the
pattern `let { a, b, ...rest } = record` — that provides compile-time guarantees
no existing mainstream language can express. Using Idris2's dependent type system,
we prove three properties of any record partition: exhaustiveness (no fields lost),
disjointness (no fields duplicated), and completeness (field counts are preserved).
We then show how Ephapax's linear types enforce single-use field consumption at the
value level, preventing a class of bugs impossible to catch with structural type
systems alone. Finally, we describe a practical ReScript PPX that implements the
verified algorithm as a compile-time code generator, making the benefits available
to the JavaScript ecosystem today.

== 1. Introduction

=== 1.1 The Problem

Modern UI frameworks (React, Solid, Svelte) rely heavily on "props spreading" —
extracting known fields from a record and forwarding the rest to a child component:

[source,javascript]
----
const { className, children, ...otherProps } = props;
return <div className={className} {...otherProps}>{children}</div>;
----

TypeScript and Flow support this via structural subtyping, but provide no proof
that the partition is correct. ReScript, despite its stronger type system, cannot
express rest patterns on records at all — a limitation that forces verbose
workarounds and brittle code.

== 2. Formal Model

=== 2.1 Record Schemas

A record schema is a list of named fields:

[source,idris]
----
Schema : Type
Schema = List Field

record Field where
  constructor MkField
  name : FieldName
----

=== 2.2 Subset and Complement

Given a schema `full` and a requested subset `sub`, the complement `rest`
contains every field in `full` that is not in `sub`:

[source,idris]
----
data Complement : (sub : Schema) -> (full : Schema) -> (rest : Schema) -> Type where
  CompNil  : Complement sub [] []
  CompKeep : NotElem f sub -> Complement sub fs rest
           -> Complement sub (f :: fs) (f :: rest)
  CompDrop : Elem f sub -> Complement sub fs rest
           -> Complement sub (f :: fs) rest
----

=== 2.3 Verified Partition

[source,idris]
----
record Partition where
  constructor MkPartition
  full           : Schema
  extracted      : Schema
  remainder      : Schema
  extractedValid : Subset extracted full
  remainderValid : Complement extracted full remainder
----

== 3. Proofs

=== 3.1 Exhaustiveness + Disjointness (ExactlyOne)

Every field in `full` lands in exactly one of `extracted` or `rest`:

[source,idris]
----
data ExactlyOne : Field -> Schema -> Schema -> Type where
  InExtracted : Elem f sub -> NotElem f rest -> ExactlyOne f sub rest
  InRemainder : NotElem f sub -> Elem f rest -> ExactlyOne f sub rest

fieldInExactlyOne : Complement sub full rest
                  -> Elem f full
                  -> ExactlyOne f sub rest
----

=== 3.2 Completeness

The extracted count plus the remainder count equals the full schema length:

[source,idris]
----
completenessProof : (comp : Complement sub full rest)
                  -> countExtracted comp + countRemainder comp = length full
----

=== 3.3 Construction

The complement is decidably computable:

[source,idris]
----
computeComplement : (sub : Schema) -> (full : Schema)
                  -> (rest : Schema ** Complement sub full rest)
----

== 4. Linear Rest-Patterns in Ephapax

=== 4.1 Syntax

[source]
----
let { className ⊸ c, children ⊸ ch, ...rest } = props
----

The `⊸` (lollipop) operator consumes the field linearly. `rest` is the
linear remainder — the type system guarantees it contains exactly the
unconsumed fields.

=== 4.2 Typing Rule

----
Γ ⊢ e : { f₁ : τ₁, ..., fₙ : τₙ }
S = { fᵢ₁, ..., fᵢₖ } ⊆ { f₁, ..., fₙ }
R = { f₁, ..., fₙ } \ S
────────────────────────────────────────────────
Γ ⊢ let { fᵢ₁ ⊸ x₁, ..., fᵢₖ ⊸ xₖ, ...rest } = e
    : Γ, x₁ :¹ τᵢ₁, ..., xₖ :¹ τᵢₖ, rest :¹ { R }
----

The superscript ¹ denotes linear multiplicity (exactly one use).

=== 4.3 Relationship to Idris2 Proofs

The typing rule above is sound because `computeComplement` constructs a
valid `Complement`, and `fieldInExactlyOne` proves the partition is
exhaustive and disjoint. The linear multiplicity annotation adds
value-level enforcement that the proof operates at the type level.

== 5. ReScript PPX (Practical Implementation)

=== 5.1 Usage

[source,rescript]
----
@rest let { className, ?children, ...otherProps } = props
----

=== 5.2 Expansion

The PPX expands to explicit field extraction at compile time:

[source,rescript]
----
let className = props.className
let children = props.children
let otherProps = {
  onClick: props.onClick,
  style: props.style,
  // ... (all fields not named above, derived from the record type)
}
----

=== 5.3 Correctness Argument

The PPX implements the same algorithm as `computeComplement`. Since
`computeComplement` is total and proven correct in Idris2, and the PPX
is a direct transliteration, the generated code is correct by construction.

== 8. Conclusion

Record subset destructuring is a common pattern that no mainstream language
handles with full compile-time safety. By combining dependent types (for
partition proofs) with linear types (for consumption guarantees), we achieve
a level of assurance that structural type systems fundamentally cannot match.
The ReScript PPX makes this practically useful today; the Ephapax language
will make it native.
